# About the Sources Directory

## Purpose

The `sources/` directory is a reserved location for caching and tracking the **source documents** and **file mappings** used by the wiki compilation system.

This directory supports the `/wiki-update` automation by maintaining:

1. **Source tracking:** Which files/docs the wiki was compiled from
2. **Change detection:** Identifying what changed since last update
3. **Attribution:** Crediting sources in wiki page frontmatter
4. **Mirroring:** Cached copies of external documentation

## File Organization

### `config.yml` (Optional)
Configure how the wiki update system scans your project.

```yaml
project:
  name: "My Project"
  language: "typescript"
  
mappings:
  architecture:
    sources:
      - "src/lib/**/*.ts"
      - "docs/architecture/"
    target: "l1-context/architecture.md"
    frequency: "weekly"
  
  conventions:
    sources:
      - "CLAUDE.md"
      - ".eslintrc.js"
    target: "l1-context/conventions.md"
    frequency: "monthly"
```

### `external/` (Optional)
Mirror external documentation sources.

```
sources/
└── external/
    ├── openapi.yml      # OpenAPI spec for API docs
    ├── db-schema.sql    # Database DDL
    └── guides/          # External API documentation
```

### `metadata/` (Internal)
Auto-generated by wiki tooling. Do not edit.

```
sources/
└── metadata/
    ├── last-update.json
    ├── file-hashes.json
    └── drift-report.json
```

## How It Works

1. **Initial Scan (`/wiki-update --full`)**
   - Scans source files per `config.yml` mappings
   - Creates hashes and metadata
   - Populates wiki pages with extracted content

2. **Incremental Updates (`/wiki-update`)**
   - Compares file hashes against `sources/metadata/file-hashes.json`
   - Identifies changed files
   - Prompts user to merge changes into wiki

3. **Drift Detection**
   - Compares wiki frontmatter `sources` against actual files
   - Flags pages with missing or outdated source references
   - Suggests re-running `/wiki-update`

## Best Practices

- **Don't edit `sources/metadata/` directly** — It's auto-managed
- **Version control:** Commit `config.yml` and `external/` to git
- **Large files:** For big specs (OpenAPI, database DDL), store in `external/` not inline
- **External docs:** Mirror external service docs in `external/` with last-fetched dates
- **Ignore:** Add `.gitignore` rule for `sources/metadata/` if it's large

## Example: Adding a New Source

To start tracking a new source file:

1. Add mapping to `sources/config.yml`:
   ```yaml
   myNewSource:
     sources:
       - "src/features/my-feature/**/*.ts"
     target: "l2-reference/features/my-feature.md"
     frequency: "per-sprint"
   ```

2. Run:
   ```bash
   npm run wiki:update --full
   ```

3. Review and merge changes into wiki

4. Commit `config.yml` to git

## Automation Hooks

The wiki system checks `sources/metadata/` to:
- Skip updates for files that haven't changed
- Show what changed since last session
- Calculate confidence scores based on file age
- Suggest next update time based on frequency
